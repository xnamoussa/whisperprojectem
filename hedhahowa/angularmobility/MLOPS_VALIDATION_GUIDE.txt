================================================================================
SASA MOBILITY - WEEK S12 MLOPS VALIDATION GUIDE
================================================================================

This guide explicitly answers each of the 7 validation criteria requested by 
your teacher. Use this to demonstrate your work.

--------------------------------------------------------------------------------
1. EXPERIMENT TRACKING (MLflow)
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Open http://localhost:5000 in your browser, select 
  "Urban_Mobility_Risk_Prediction" on the left, and click the "Runs" tab.
- EXPLANATION: We implemented MLflow to track every training execution and perform True Model Comparison. 
  You can see runs logged for different algorithms (Logistic Regression, Random Forest, Gradient Boosting)
  with their specific hyperparameters, "intuition" tags, and evaluation metrics (Accuracy, F1 Score). 
  Clicking a run shows the "Artifacts" section where the physical model is saved.
- HOW TO PROVE IT: Open MLflow (http://localhost:5000). Show the 3 runs for the different algorithms. 
  Check the boxes next to them and click "Compare" to show the teacher how you evaluate which model 
  is best based on the F1-Score.

--------------------------------------------------------------------------------
2. AUTOMATED TRAINING PIPELINE
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Open the file `ml_service/train_pipeline.py`.
- EXPLANATION: This script represents our end-to-end reproducible pipeline. 
  Without manual intervention, it automatically generates/loads data, scales 
  numeric values, encodes categorical values (preprocessing), iterates over 
  multiple classification algorithms, evaluates their metrics, automatically 
  identifies the best performer, and saves the final model to both MLflow and 
  the local disk.
- HOW TO PROVE IT: Open a terminal in the ml_service folder (or in Docker) and run 
  `python train_pipeline.py`. Show the teacher the console output as it trains the 3 models 
  automatically and announces the "Meilleur Modèle" at the end without you doing anything.
docker exec mobility-ml-api python train_pipeline.py
  

--------------------------------------------------------------------------------
3. MODEL MANAGEMENT
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Check the `ml_service/models/` and `ml_service/mlruns/` folders.
- EXPLANATION: Every time the pipeline runs, a new versioned folder is created 
  inside `mlruns/` keeping previous versions completely accessible via MLflow. 
  The "best" or latest production model is explicitly exported to 
  `models/risk_model.joblib` for the API to consume.
- HOW TO PROVE IT: In MLflow, click on a run and scroll down to "Artifacts" to show the saved model. 
  Then, open your VS Code file explorer and show the teacher the `ml_service/models/risk_model.joblib` 
  file, proving the API has physical access to the winning model.

--------------------------------------------------------------------------------
4. MODEL SERVING (API)
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Open http://localhost:8001/docs in your browser.
- EXPLANATION: We built a functional FastAPI service that loads the saved model. 
  The `/predict` endpoint accepts a JSON payload of urban mobility data (city, 
  traffic, NO2 levels, etc.) and returns a predicted risk level (High/Low) 
  along with the model's confidence probability. You can test it directly on 
  the /docs page.
- HOW TO PROVE IT: Open http://localhost:8001/docs. Click on the `/predict` endpoint, click "Try it out", 
  enter the following JSON data, and hit Execute. Show the teacher the API returning a 200 OK 
  response with the predicted Risk Level (High/Low).

  ```json
  {
    "city": "Paris",
    "station_type": "Metro",
    "lat": 48.8,
    "lon": 2.3,
    "connections": 5,
    "daily_traffic": 45000,
    "no2": 38.5
  }
  ```

--------------------------------------------------------------------------------
5. CONTAINERIZATION
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Open the `docker-compose.yml` file in the root folder.
- EXPLANATION: The entire architecture is fully containerized. We have 
  independent Docker services for the Backend, Frontend, MLflow server, and 
  the FastAPI Model Serving API. They communicate seamlessly over a shared 
  Docker network, ensuring the app runs flawlessly anywhere.
- HOW TO PROVE IT: Open Docker Desktop or run `docker ps` in your terminal. Show the teacher that 
  `mobility-ml-api` and `mobility-mlflow` are running as completely independent containers alongside 
  your Angular frontend and Django backend.

--------------------------------------------------------------------------------
6. CODE QUALITY
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Across the `ml_service/` folder.
- EXPLANATION: The code is highly modular. The ML training (`train_pipeline.py`) 
  is strictly separated from the API serving (`main.py`). The API uses 
  Pydantic models (`PredictionInput`) to strictly validate incoming data types, 
  preventing major runtime errors and keeping the code clean and structured.
- HOW TO PROVE IT: Open `ml_service/main.py` and point out the `PredictionInput` class. 
  Explain to the teacher how Pydantic guarantees that the API will never crash due to bad data types 
  (e.g., if someone accidentally sends a string instead of a number for traffic).

--------------------------------------------------------------------------------
7. WEB APP INTEGRATION
--------------------------------------------------------------------------------
- WHERE TO FIND IT: Open the Angular App at http://localhost:4200. Go to the 
  Dashboard and click the "IA Industrialisée" tab (Rocket/Globe Icon).
- EXPLANATION: We built a dedicated UI for the MLOps inference. When a user 
  adjusts the sliders (like Daily Traffic or NO2) and clicks the run button, 
  the Angular app sends an HTTP request to our FastAPI service. The API queries 
  the best-performing model (e.g., Gradient Boosting) and passes the prediction 
  back to the UI, proving the full end-to-end pipeline works flawlessly.
- HOW TO PROVE IT: Open the Angular app (http://localhost:4200), go to the "IA Industrialisée" tab. 
  Change the sliders (e.g., put NO2 and Traffic very high) and click Predict. Show the result changing 
  live on the screen, proving the UI successfully talks to the Python API.
